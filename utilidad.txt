2) ¿Qué es IoC (Inversión de Control)?
Concepto simple:
Normalmente, cuando un objeto A necesita usar un objeto B:
    1. A crea B (control está en A)
    2. Con IoC: A recibe B ya creado (el control de crear B está INVERTIDO - no lo hace A)
Ejemplo concreto de TU proyecto:
Sin IoC (si hicieras todo en el Servlet):
java
public class ProcesarApuestaServlet {
    // El Servlet hace TODO
    public void procesar() {
        // 1. Conexión a DB
        Connection conn = DriverManager.getConnection(...);
        // 2. Ejecuta query
        // 3. Cálculos de apuestas
        // 4. Actualiza saldo
        // 5. Cierra conexión
    }
}
Con IoC (como lo tienes):
java
public class ProcesarApuestaServlet {
    public void procesar() {
        // 1. Pides el servicio (UsuarioService) ← NO lo creas tú
        UsuarioService usuarioService = new UsuarioService();  // ← Aquí el servicio SÍ crea sus propias dependencias
        
        // 2. Usas el servicio
        usuarioService.getDineroPorIdUsuario(...);
        // El control de CÓMO se obtiene el dinero está INVERTIDO: no lo controla el Servlet
    }
}

Mecanismo de sesión HTTP:
text
Primera petición (Login):
Usuario → POST /login → Servlet
                     ↓

Usuario → GET /billetera → Servlet

                     ↓

                     ↓

2) PARÁMETROS DE RUTA (ROUTE PARAMS) - NO TIENES EN FORMA NATIVA
¿Por qué no tienes Route Params?
Tu proyecto usa el patrón clásico de Servlets donde las URLs se mapean así en web.xml:
xml
<servlet-mapping>
    <servlet-name>ApuestaServlet</servlet-name>
    <url-pattern>/Apuesta</url-pattern>  ← URL fija, sin parámetros en ruta
</servlet-mapping>
Las tecnologías que SÍ usan Route Params:
    • Spring MVC: @GetMapping("/partidos/{id}")
    • JAX-RS (Jersey): @Path("/partidos/{id}")
    • Express.js: app.get('/partidos/:id', ...)
Pero tienes un PATRÓN SIMILAR: ApuestaServlet.java
java
// En ApuestaServlet.java (debe existir)
public class ApuestaServlet extends HttpServlet {
    
    @Override
    protected void doGet(HttpServletRequest request, HttpServletResponse response) 
            throws ServletException, IOException {
        
        // ✅ SIMULACIÓN DE PARÁMETRO DE RUTA mediante Query Param
        String idParam = request.getParameter("id");
        // Esto parece query param pero actúa como route param
        
        if (idParam != null && !idParam.trim().isEmpty()) {
            try {
                int idPartido = Integer.parseInt(idParam);
                
                // Obtener partido por ID
                PartidoRepository repo = new PartidoRepository();
                Partido partido = repo.obtenerPorId(idPartido);
                
                if (partido != null) {
                    request.setAttribute("partido", partido);
                    request.getRequestDispatcher("WEB-INF/jsp/apuesta.jsp").forward(request, response);
                } else {
                    // Error: partido no encontrado
                }
                repo.close();
                
            } catch (NumberFormatException e) {
                // Error: ID inválido
            }
        } else {
            // Error: falta ID
        }
    }
}

    • Técnicamente es query param (?id=7)
    • Pero conceptualmente actúa como route param porque identifica el recurso

Para responder:
1. Parámetros de Consulta (Query Params) - SÍ tienes:

2. Parámetros de Ruta (Route Params) - NO tienes nativos:
    • Tu proyecto usa Servlets clásicos que no soportan route params como /{id}
    • Pero simulas el comportamiento con query params:

Diferencia clave:
    • Query Params: ?clave=valor&clave2=valor2 → para filtros, opciones, configuraciones
    • Route Params: /recurso/{id} → para identificar recursos específicos (en tu caso, lo simulas con query params)
En la práctica de tu proyecto:
    • Usas query params reales para: modo admin (?admin=true), acciones (?eliminar=5)
    • Usas query params como route params para: identificar partidos (?id=7)
Esta es una limitación/rasgo del stack tecnológico elegido (Servlets Jakarta EE). Frameworks modernos como Spring Boot permiten route params nativos, pero para un proyecto académico con Servlets, tu aproximación es válida y común.
